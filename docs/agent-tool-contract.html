Agent Tool Contract v1

让工具更容易被 Agent 正确调用。

这不是一套更长的 API 文档，而是一份把“意图、参数、权限、失败和输出”说清楚的协作协议。Forge 用它连接 GitHub 与 GitLab，同时保护 Agent 的上下文窗口。

一句话理解 Agent-friendly 不等于接受模糊输入。它意味着：可推导的信息自动推导，必须明确的意图强制明确，失败时告诉 Agent 下一步，返回时只给真正需要的信息。

我们先解决三个真实问题

工具调用失败通常不是 Agent 不会工作，而是工具把太多歧义、平台差异和原始数据交给了它。

01 / 歧义

占位字段也算“传了”

id: ""、ids: [0] 仍会触发互斥校验。模型看见一个大参数袋，很容易把所有字段都填上。

02 / 差异

GitHub 与 GitLab 不是同一个 API

Release、pipeline、milestone、merge 的字段和能力不同。静默丢弃字段会制造“工具成功、语义错误”。

03 / 上下文

列表会吞掉注意力

几十条 issue 的完整正文、用户对象和 URL 会挤走真正需要的决策信息，Agent 反而更难检索。

契约由六层组成

Schema 只是第一层。每一层都在减少一种具体的调用错误。

01

意图

useWhen、doNotUseWhen 和最小示例告诉 Agent 什么时候调用，什么时候先做别的事。

02

参数

封闭、按操作区分的 schema。list、get、get_many 不共享一份模糊参数袋。

03

副作用

每个工具标记为 read、routine_write、sensitive_write 或 destructive，写入动作继续经过策略门禁。

04

幂等与重试

契约声明是否可安全重试、最多几次，以及确定性失败是否拒绝完全相同的调用。

05

平台能力

能翻译就翻译，不能保持语义就返回 unsupported_capability，绝不静默忽略请求。

06

输出预算

列表、详情、错误和 mutation receipt 都有字节预算。结果是投影，不是 provider 原始响应。

一次调用如何被保护

Agent 只需要表达任务意图；平台细节和安全边界由工具链逐层处理。

01

表达意图

选择 operation 或 action，不填无关占位字段。

02

规范化

把等价写法统一，例如把数字字符串转成数字。

03

校验

封闭 schema 拒绝互斥字段、空占位和不支持的组合。

04

门禁

检查仓库信任、token、策略、平台 capability 和确认要求。

05

投影结果

返回稳定、紧凑、有预算的 receipt 或结构化错误。

从“参数大杂烩”到明确操作

显式 operation 让模型先做一个小而重要的决定：我要列出、读取一个，还是读取一组？

容易出错的 payload
{
  "resource": "issue",
  "id": "",
  "ids": [0],
  "state": "opened",
  "search": "x",
  "limit": 50
}
明确且最小的 payload
forge_query · list
{
  "operation": "list",
  "resource": "issue",
  "state": "open",
  "search": "x",
  "limit": 10
}
关键取舍：工具不会替 Agent 猜测 id 还是 ids。自动猜测可能执行错误读取；明确失败并阻止原样重试，才会迫使调用真正变正确。

完整演练：从当前仓库找到一个 issue

下面是一条推荐的读流程。每一步都只做一个决定，结果也只返回下一步需要的信息。

1

确认仓库与平台

Agent 不填写 remote、project 或 provider。当前 Git worktree 的 remote 是确定事实，工具自动解析它。

forge_context
{}
2

用服务端搜索缩小候选

先 list，不要一次读取所有正文。默认只取 10 条，必要时再用 cursor 翻页。

forge_query · list
{
  "operation": "list",
  "resource": "issue",
  "state": "open",
  "search": "登录失败",
  "limit": 10
}
3

读取一个明确详情

拿到候选 number 后，才请求单条详情。详情预算更高，但仍然有上限。

forge_query · get
{
  "operation": "get",
  "resource": "issue",
  "id": 42
}
4

需要写入时再进入 mutation

写操作使用明确 action、领域标识和最小字段，并经过 policy gate 与确认要求。

forge_mutate · update
{
  "resource": "issue",
  "action": "update",
  "issueNumber": 42,
  "set": { "state": "closed" },
  "requestId": "close-issue-42"
}
读写边界：capability 的 supported 只说明读取端点可访问，不等于 token 拥有创建、修改、合并、重跑或发布权限。写权限要由真实写请求和策略门禁验证。

参数决策树：先问自己是哪一种任务

Agent 友好的工具不要求模型记住几十个端点，而是把任务压缩成几个稳定的选择。

我要找候选→operation: list

提供资源和筛选条件，例如 state、search、limit。不要提供 id 或 ids。

我知道一个编号→operation: get

提供单个领域标识，例如 issueNumber 或查询中的 id。不要把它包装成数组。

我知道一组编号→operation: get_many

提供有限的 ids，适合已经从 list 结果中选出的少量资源，不适合全库导出。

我要改变远端→action + requestId

先确认资源、action、真正修改的字段和重复执行策略，再接受写入的确认与权限检查。

平台映射：Agent 不需要学习两套字段

公开契约保持领域语言，适配器负责把同一意图翻译成 GitLab 或 GitHub 的请求。

Agent 看到的概念GitLab 适配GitHub 适配不能静默处理时
bodydescriptionbody保持正文语义，不丢内容
sourceBranch / targetBranchsource_branch / target_branchhead / base字段无法表达时返回 capability 错误
milestone number先用 IID 解析全局 ID使用 GitHub milestone number找不到目标返回 not_found
release tag解析 release 并映射版本字段按 tag 解析 release IDdraft、prerelease 等不支持语义明确拒绝
job retry调用 GitLab retry endpoint调用 Actions job rerun endpoint没有等价动作返回 unsupported_capability
pipeline state使用平台状态字段failed → failure、running → in_progress 等无法保持状态含义时不猜测

这张表的重点不是记忆映射，而是理解边界：通用参数只有在两个平台都能保持相同语义时才会进入公共契约。

失败不是一堵墙，而是恢复信号

结构化错误把“发生了什么”和“下一步做什么”分开，让 Agent 不需要从自然语言里猜策略。

结构化错误
{
  "ok": false,
  "error": {
    "code": "rate_limited",
    "retryable": true,
    "message": "Try again after cooldown",
    "details": { "retryAfterSeconds": 30 }
  }
}
错误类型Agent 动作
validation_error修正参数
unsupported_capability换操作或停止
forbidden检查权限，不盲重试
rate_limited有限重试
duplicate_failed_call改变参数或停止

第一次失败

工具保存调用指纹和失败类型，告诉 Agent 这是参数问题、权限问题还是瞬时问题。

原样重试

如果失败是确定性的，完全相同的规范化参数会被阻止，避免浪费请求和上下文。

真正修正

参数改变、用户开启新一轮，或瞬时失败在额度内重试，调用链重新获得机会。

重试状态：什么会重试，什么不会

重试不是统一的“再来一次”。工具根据错误性质和规范化后的调用指纹决定下一步。

状态例子相同参数再次调用正确恢复方式
确定性失败schema 校验失败、forbidden、unsupported_capability阻止修正参数、换操作或补充权限
瞬时失败rate_limited、短暂网络错误有限允许遵守 retry policy 和退避信息
参数已变化从 id: "7" 改为合法的 id: 7允许重新进入规范化与校验流程
用户开启新一轮用户提供新要求或外部状态发生变化允许旧失败边界不再阻塞新任务
调用已成功同一资源之前已经成功读取或写入重置后续新失败重新开始计数
指纹很重要：工具先执行 prepareArguments，再进行运行时校验和属性排序。这样 {"id":"7"} 与 {"id":7} 如果最终表示同一个调用，就不能靠改变 JSON 表面写法绕过失败保护。

输出投影：为什么不返回原始 provider 对象

返回值要服务于下一步决策，而不是复制平台 API 的所有字段。

不适合进入 Agent 上下文
{
  "id": 42,
  "title": "登录失败",
  "user": {
    "avatar_url": "...",
    "followers_url": "...",
    "events_url": "...",
    "public_repos": 18
  },
  "repository_url": "...",
  "timeline_url": "...",
  "body": "非常长的正文..."
}
适合决策的紧凑投影
{
  "ok": true,
  "resource": "issue",
  "number": 42,
  "title": "登录失败",
  "state": "open",
  "author": "frelion",
  "webUrl": "https://..."
}
  • 列表只保留定位、比较和下一步读取需要的字段。
  • 单条详情通过显式 get 请求获取，正文仍受字节预算限制。
  • mutation journal 保存 compact receipt，不保存 token、headers 或 provider 原始 body。
  • 错误 details 使用白名单，循环对象、BigInt、超大字符串和 secret-like 字段不会破坏上下文。

上下文窗口是正式资源

Forge 不把“更多原始数据”误认为“更多检索能力”。它先返回定位信息，再按需读取详情。

10列表默认条数
50列表最大条数
384 B列表单条正文预览
16 KiB单条详情正文
8 KiB批量详情单条正文
64 KiBForge 单次最终结果

推荐检索路径

list 用于定位，配合 search 缩小候选集；使用 cursor 翻页；确认目标后用 get 读取单条详情；只有已经明确知道一组小 ID 时才用 get_many。

mutation 只返回 id、number、state、webUrl 等 receipt，不把完整 GitHub/GitLab 响应写进对话或 journal。

新增一个工具时，逐项检查这份清单

这份清单把“看起来能用”变成可以 review、可以测试的契约。

调用前

  • 工具名称是否表达一个明确动作，而不是一个万能入口？
  • 是否有 useWhen、doNotUseWhen 和最小合法示例？
  • 是否按 operation/action 拆分了互斥参数？
  • 是否关闭了未知字段，并拒绝空字符串、零值和空数组占位？

执行中

  • 是否使用领域标识，而不是暴露 provider 内部 ID 规则？
  • GitHub 与 GitLab 的字段是否能保持同一语义？
  • 不支持的操作是否返回明确的 unsupported_capability？
  • 写操作是否经过 effect、policy、确认和 request ID 门禁？

执行后

  • 成功结果是否是紧凑 receipt，而不是原始 provider body？
  • 列表、详情、错误和 journal 是否都有独立预算？
  • 错误是否包含稳定 code、retryable 和可恢复 details？
  • 同一确定性失败是否会阻止无变化重试？

测试中

  • 是否测试真实 Agent loop，而不只测试函数返回值？
  • 是否覆盖错误发生在 prepare、execute、after hook 三个入口？
  • 是否测试空响应、非法 JSON、401/403/404/5xx 和网络错误？
  • 是否用大列表和大错误验证上下文预算没有被突破？

最后记住这五条

能推导的，自动推导

当前 Git remote 决定 repository、host 和 provider。

必须明确的，强制明确

operation、action、资源标识和真正要更新的字段不能靠猜。

不能做的，明确拒绝

平台不支持的语义返回 capability 错误，不静默降级。

会失败的，说明恢复

稳定 code、retryable 和 bounded details 让 Agent 采取正确动作。

会变大的，先做投影

列表和错误都有字节预算，详情通过显式操作按需读取。

要验证的，端到端验证

不仅测 schema，还测试真实 Agent loop 的重试、错误和上下文边界。
